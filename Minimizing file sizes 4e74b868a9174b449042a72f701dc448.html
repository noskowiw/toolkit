Minimizing file sizes

In short:

  • When working files that are meant to be loaded frequently (like pictures on a website or files containing data necessary to perform some calculation) it�s important to keep them as small as they possibly can be.
  • interpreted files (like .js, .css, .json) that are not meant to be edited further should always be compressed.
  • In some cases it�s more energy efficient to perform extra processing on a smaller file than to load a larger file from memory, thou striking a balance between extra processing and smaller file sizes should be made on case by case basis.

Transferring data over the network or loading them from a hard drive always comes with an inherent energy cost (more on that in {{networking}}). That�s why from a sustainability standpoint it�s important to keep the size of these files as small as it possibly can be.

For the most part working with compressed file formats (for example .jpg or .png for pictures) is more beneficial from the energy efficiency standpoint.

Using more complex compression methods may cause enough computational overhead to outweigh the advantages gained by reducing file size. In those cases the right balance between size and computation can be found with the help of testing and {{benchmarking}}.

Make sure that you�re storing only the data you need. If you�re dropping a significant amount of the files contents during parsing then it�s probably a good idea to refactor the file you�re loading.

Interpreted text files that are not meant to be human readable (like .js, .css or .json files) can also be compressed or minified - in this case this means that all the whitespace characters are removed.
